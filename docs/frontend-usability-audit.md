# 前端易用性审计记录

本记录只整理改进方向，不直接修改前端实现。

## 审计方式

- 实际运行地址：`http://localhost:8787`
- 浏览器体验视口：
  - 桌面：1280 x 720
  - 移动端：390 x 844
- 体验页面：
  - `/` 首页计算器
  - `/drive-discs.html` 驱动盘仓库
  - `/calculate.html` 驱动盘优化器
  - `/accounts.html` 账号页
  - `/maintenance.html` 维护页
- 实际操作：
  - 首页点击“开始计算”，查看结果位置和反馈。
  - 打开“选择技能倍率”“选择 Buff”“选择驱动盘”等弹窗。
  - 在 Buff 弹窗勾选一条队友 Buff，观察反馈。
  - 驱动盘仓库打开“新增驱动盘”弹窗并滚动到底部。
  - 优化器打开页面、运行一次计算、取消计算、打开“计算配置”弹窗。
  - 移动端查看首页和优化器首屏。

## 总体判断

项目已经具备很完整的计算能力，但前端当前的主要问题不是“缺功能”，而是功能密度过高以后，用户很难判断下一步应该看哪里、点哪里，以及某次操作是否已经生效。尤其是首页计算结果、优化器长任务、长表单弹窗和维护页，已经到了需要重排信息架构的阶段。

建议优先处理三件事：

1. 让主操作和结果反馈直接相连。
2. 让长页面、长弹窗和三栏布局的滚动关系显性化。
3. 把危险操作、长配置和调试信息从默认主路径中移走。

## 高优先级改进

### 1. 首页“开始计算”后结果不可见，容易误以为没有反应

桌面端点击首页“开始计算”后，伤害已经算出，`#damageFinalValue` 更新为结果，但“伤害白盒”在首屏下方较远位置，当前视口看不到。按钮本身也没有跳转、聚焦、成功反馈或结果摘要。

涉及位置：

- `frontend/index.html`：顶部按钮和结果区分离。
- `frontend/app.js`：`calculateBtn` 点击后只执行计算和状态更新，没有把用户带到结果区。

建议：

- 计算完成后滚动或聚焦到“伤害白盒”，或者在顶部/hero 区展示一个轻量结果摘要。
- “开始计算”在首页语义可以改为“重新计算”或“更新结果”，因为页面默认已经有大量初始结果。
- 首页 hero 里的“目标函数 / 简单评分”容易和优化器混淆，建议换成当前伤害目标摘要，例如“直伤 · 100% 倍率 · 期望”。

### 2. 优化器三栏固定一屏且隐藏滚动条，很多内容像是消失了

桌面端优化器在宽屏下把主内容固定成 `100vh`，左/中/右三栏分别内部滚动，并隐藏滚动条。实际体验中，计算设置下方的主词条限制、结果进度、保存套装等内容不明显，用户不知道某一栏还能继续滚。

涉及位置：

- `frontend/styles.css` 中 `@media (min-width: 1241px)` 的 `.optimize-content`、`.optimizer-workspace`、`.optimizer-left/.optimizer-center/.optimizer-right`。
- 当前 CSS 明确设置了 `overflow-y: auto` 和 `scrollbar-width: none`。

建议：

- 不要隐藏优化器三栏滚动条，至少保留细滚动条或底部渐隐提示。
- 将优化器重排为更明确的步骤：
  1. 角色与音擎
  2. 局内/Boss 条件
  3. 计算目标与约束
  4. 运行与结果
- 让“开始计算”和“计算目标/约束/复杂度预估”靠近，而不是只放在页面最顶端。
- 计算运行时将进度摘要固定在顶部或结果栏首屏，避免用户启动后看不到进度。

### 3. 优化器取消反馈不够干净

实际点击“取消计算”后，界面进入“取消中/正在取消”，但右上仍可见“取消计算”按钮；等待一段时间后才显示“已取消”。这个阶段用户会不确定是否需要再点一次。

涉及位置：

- `frontend/calculate.js`：`setOptimizeButtonRunning` 会在运行态显示取消按钮，`cancelActiveOptimization` 中又把主按钮改成“取消中”。

建议：

- 用户点击取消后，只保留一个明确的禁用状态，例如主按钮显示“正在取消...”，取消按钮隐藏或禁用并改为“已发送取消请求”。
- 如果后端取消需要等待 chunk 结束，前端应该明确说明“正在等待当前批次结束”。
- 进度条旁显示最近一次后端响应时间，避免长时间无变化时显得卡住。

### 4. 长弹窗的保存/关闭会滚出视野

“新增驱动盘”弹窗内容很长，滚到底部编辑主词条、副词条时，顶部的“保存/关闭”已经不可见。用户填完后需要滚回顶部才能保存。

涉及位置：

- `frontend/drive-discs.html`：驱动盘编辑弹窗。
- `frontend/styles.css`：`.disc-modal` 使用内部滚动，`.modal-actions` 只是普通 flex 区域。

建议：

- 给长弹窗加 sticky header 或 sticky footer，让保存、取消、删除始终可见。
- 底部重复一组“取消 / 保存”操作也可以，但不要和顶部按钮产生不同状态。
- 弹窗内可加分段锚点：套装、位置、品质等级、主词条、副词条、预览。

### 5. 危险操作和原生浏览器弹窗需要收敛

账号新增/改名使用 `window.prompt`，账号删除和驱动盘删除/清空/导入同步使用 `window.confirm`。这些原生弹窗打断体验，也无法承载足够清楚的风险说明、数量预览和校验。

涉及位置：

- `frontend/accounts-page.js`：新增、改名、删除账号。
- `frontend/drive-discs.js`：导入时删除缺失、清空仓库、删除驱动盘、删除套装预设。
- `frontend/calculate.js`：保存优化器结果为套装时使用套装名称 prompt。

建议：

- 改为项目内统一确认弹窗。
- 危险操作展示影响范围，例如会删除几个驱动盘、几个套装预设、当前账号是谁。
- “删除不存在的驱动盘”不要作为导入按钮旁的常驻 checkbox，建议放进导入流程的高级选项或二次确认里。

## 页面级改进建议

### 首页

问题：

- 配置区和结果区同时铺开，用户需要横向扫很多内容。
- “局外面板”“局内面板”“伤害白盒”“攻击力明细”“原始结果”全部在主路径中，结果层级偏重。
- 技能选择弹窗没有搜索，也没有明显展示当前已选倍率。
- Buff 弹窗勾选单条 Buff 后，计数文案显示“已选择 1/2 名角色”，对正在勾选单条效果的用户不够贴切。

建议：

- 结果区顶部做一个固定摘要：最终伤害、关键乘区、当前技能、启用 Buff 数。
- “原始结果”默认折叠到调试区，不作为普通用户主路径。
- 技能倍率弹窗增加搜索、当前选择标记和“只看当前大类”开关。
- Buff 弹窗同时显示“队友 1/2、Buff 1 条”，并给 tab 加数量。

### 驱动盘仓库

做得好的地方：

- 表格模式适合 425 条驱动盘的扫描。
- 筛选、槽位 tab、排序都在首屏，批量浏览效率不错。

问题：

- “删除不存在的驱动盘”离“导入”太近，含义依赖用户理解扫描器同步逻辑。
- 槽位 tab 只显示数字，不显示每个槽位数量。
- 空态文案仍有 `No Data`，中英文混用。
- 新增/编辑驱动盘弹窗很长，保存按钮不固定。

建议：

- 将导入做成一个弹窗流程：选择文件 -> 预览导入数量 -> 可选同步删除 -> 确认。
- 槽位 tab 显示 `1号位 80` 这样的数量。
- 统一空态中文文案，例如“暂无数据”。
- 表格头在滚动容器中保持 sticky，便于长列表横向理解。

### 驱动盘优化器

问题：

- 桌面三栏各自滚动且滚动条隐藏。
- 移动端“开始计算”出现在所有计算设置之前，用户可能没看到算法、套装、主词条限制就启动巨大任务。
- 默认循环摘要过长，计算配置弹窗里事件列表文本密度高。
- 进度信息在结果卡里，不一定位于启动后用户正在看的区域。

建议：

- 移动端把“开始计算”移到“计算设置”之后，或在点击前弹出复杂度预览。
- 默认循环摘要压缩成 chips：直伤 xN、紊乱 xN、碎冰 xN，再允许展开详情。
- 计算配置弹窗左侧事件列表显示短标题，右侧详情再展示完整路径。
- 运行中把复杂度、速度、预计组合数做成固定运行条。

### 账号页

问题：

- 新增/改名/删除使用原生弹窗。
- 删除账号虽然有确认，但缺少项目内一致的危险操作样式。

建议：

- 新增/改名改为轻量表单弹窗。
- 删除账号弹窗展示该账号下驱动盘、套装预设、导入记录数量。
- 当前账号可以更明显地展示“正在使用”，并解释切换会影响哪些页面。

### 维护页

问题：

- 单个角色资料编辑页高度超过 3500px，字段很多。
- 保存按钮在顶部，长时间编辑到底部后不在视野内。
- JSON 保存预览对维护者有用，但占据主编辑路径。
- tab 很多，移动端和窄屏上会变成多行，定位成本上升。

建议：

- 增加 sticky 保存条：未保存变更、校验状态、保存、撤销本次修改。
- 每类资料内部增加二级分段导航，例如基础信息、面板、优先驱动盘、默认计算、Buff。
- JSON 预览默认折叠，或放入右侧/底部调试抽屉。
- 保存失败时显示顶部错误汇总，并能跳到对应字段。

## 视觉和文案统一

建议统一处理这些小问题：

- `No Data` 改为中文空态。
- `Buff`、`Boss`、`totalFinalDamage` 这类技术或英文标签只保留在调试视图，普通视图使用中文。
- “计算”导航名和“首页计算器”容易混淆，可以考虑把优化器导航命名为“优化器”。
- “同步首页”按钮语义不清，建议改成“套用首页设置”或“从首页同步”。
- 图标按钮目前有些使用 `+`、`×`，建议加可见 tooltip，并统一按钮尺寸。

## 可快速落地的改动清单

1. 首页计算完成后滚动到“伤害白盒”或展示顶部结果摘要。
2. 取消优化器三栏的隐藏滚动条。
3. 长弹窗增加 sticky 操作栏。
4. “删除不存在的驱动盘”移入导入流程二次确认。
5. 替换账号和套装命名的 `window.prompt`。
6. 优化器移动端把“开始计算”移动到计算设置之后。
7. 将 `No Data`、`totalFinalDamage` 等非用户语言收敛到中文或调试模式。
8. 维护页增加 sticky 保存条和错误汇总。

## 后续验证建议

- 用 1280 x 720、1440 x 900、390 x 844 三个视口回归检查。
- 每次修改后至少走一遍：
  - 首页计算 -> 看到最终伤害。
  - 首页选择技能 -> 回填倍率。
  - Buff 弹窗勾选/取消 -> 局内面板变化。
  - 驱动盘新增弹窗滚动到底部 -> 保存仍可见。
  - 优化器开始计算 -> 进度可见 -> 取消可见且状态一致。
  - 账号新增/改名/删除确认。
  - 维护页编辑字段 -> 校验失败能定位字段。
