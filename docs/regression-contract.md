# 长期回归契约

本文档记录 Vue、共享核心、Node 服务和静态发布必须长期保持的用户语义。界面布局与内部实现可以演进，但以下数据、计算、持久化和兼容入口不得静默改变。

## 架构边界

- `core/` 只承载可跨平台复用的计算、规范化、校验和优化搜索逻辑，不直接依赖 DOM、浏览器存储或 Node 文件系统。
- `webapp/src/runtime/` 提供 catalog、IndexedDB/localStorage、扫描器桥接和浏览器 Worker 等平台适配。
- `backend/` 提供 Node API、文件存储适配和静态产物服务；同一业务规则不得在后端和浏览器各维护一份。
- 浏览器重优化默认在本地 Worker 中运行；公开部署不得把用户仓库上传到共享服务端执行重计算。

## 选择、计算与 Buff

- 默认角色来自可展示 catalog；合法的旧选择应尽力恢复。角色切换后，推荐音擎、核心技、影画、默认循环和默认 Buff 必须按新角色重新解析。
- 音擎改装等级和角色影画等级必须夹紧到合法范围；没有核心技等级的角色允许使用 `none`。
- 计算支持单事件直伤、贯穿、异常、紊乱、自定义事件和角色 `defaultCalculationConfig`；影画变体使用不超过当前影画的最高已配置等级。
- 伤害事件的 `damageRatioPct` 必须同时作用于白盒、快速总伤、编译评分和固定套装优化路径；普通事件管理不得显示该技术比例输入，但维护端、存量 payload 和计算语义必须保留。`anomalyVariant: polarizedAssault` 必须显示为“极性强击”，`damageBasis: anomalyProficiency` 的直伤不得随攻击力变化。
- 技能、贯穿、异常、紊乱和技能组引用必须拥有事件级 `stunned`，缺失时默认 `true`。技能组引用展开时必须覆盖全部组内事件；普通、白盒、快速总伤、compiled、dense、固定评分和优化器必须读取同一事件状态。未失衡事件不应用普通失衡易伤，但仍应用 `stunDmgMultiplierBonusAlways`。
- `stunDmgMultiplierBonusCapAlways` 表示捕获的失衡易伤“加成上限”而非固定增伤：匹配事件无论是否失衡，都按 `min(目标配置失衡倍率 + 失衡倍率加算, 1 + 上限)` 计算。叶瞬光核心被动只定向明心境内招式，基础上限 110%；4影必须通过 Buff 修饰将同一规则提高到 200%，核心被动未启用时4影不得单独生效。普通、白盒、compiled、dense、固定评分和优化器必须保持一致。
- 爱丽丝额外能力必须在局内面板按 `floor(max(floor(异常掌控) - 140, 0) × 1.6)` 增加异常精通；输入异常掌控和转换结果分别向下取整。核心被动必须复用通用 `disorderBaseMultiplierBonus`，按每层 18%、默认及最高 10 层建模，不得重新引入物理异常剩余时间或物理紊乱上限专用类型。普通路径、编译路径和优化器必须给出一致结果。
- 异常掌控百分比和固定点数不得共用累加桶：核心技固定加成属于角色白值，局外 `anomalyMastery` 百分比乘白值，局内百分比乘局外面板，`anomalyMasteryFlat` 最后相加。法厄同之歌两件套固定提供 8%；满核心技爱丽丝搭配 30% 六号位时必须得到内部值 `195.96`、界面值 `195`。额外能力转换时必须先按 `floor(195.96) = 195` 取整；精 1 十方锻星生效后的 `255.96` 必须转换为 `184` 异常精通，使截图方案的局内异常精通为 `336 + 184 = 520`。普通、prepared、summary、indexed、dense、固定评分、浏览器 Worker 和严格精确优化器必须保持一致。
- 事件管理的目标事件列表必须完整换行展示可读摘要；直伤/贯穿技能摘要只显示招式和倍率行，次数在独立摘要中展示。列表与详情都必须显示当前事件倍率：技能使用当前等级倍率，属性异常计入触发次数，紊乱计入原异常、紊乱类型和已流逝时间，存量事件比例折算进读数，但事件次数和最终伤害乘区不得混入。调整相关字段后倍率必须实时更新；技能组不汇总混合倍率，但组内预览必须逐项显示子事件倍率。技能组事件编辑器必须完整显示次数步进器和当前技能组名称，不得因字段过窄而裁切。最大化单技能、贯穿或异常伤害时不得从左侧新增、复制或删除事件，具体目标只从右侧编辑。
- 紊乱 `elapsedSeconds` 的编辑步长和计算精度必须读取所选异常目录的 `tickIntervalSeconds`：灼烧、侵蚀为 `0.5` 秒，感电、畏缩、霜寒、烈霜霜寒为 `1` 秒。任意输入按最近的对应间隔归一化，例如 1 秒类异常的 `4.5` 秒归一化为 `5` 秒；事件存储不得按基础时长截断，因为最终计算可能叠加异常持续时间延长。目录倍率预览仍按基础持续时间夹紧，最终计算再按“基础持续时间 + 当前命中的 `anomalyDurationBonusSeconds`”夹紧并重算剩余时间、跳数和倍率。工作台、管理员默认循环、维护保存、白盒、快速总伤、compiled、dense、固定评分和优化器必须使用同一结果；未知旧异常回退 `0.5` 秒。
- 简队友 Buff 只收录 F 级核心被动、2 影和 4 影。核心只把畏缩从 10 秒延长到 15 秒，满时长倍率必须为 `562.5%`，输入已流逝 `0.5` 秒时必须为 `555%`；不得影响其他紊乱或普通强击 `713%`。核心异常暴击率使用 `clamp(40 + 简异常精通 × 0.16, 0, 100)%`，输入默认 375、允许向下调整；核心与 2 影合计 100% 异常暴击伤害，2 影 15% 防御无视只命中强击。4 影只令属性异常增伤乘区为 `1.18`，紊乱增伤乘区保持 `1.0`。
- 管理员默认计算的右侧事件详情必须以正常文本色展示中文静态值，包括只读的“是否失衡”；不得用低对比度禁用控件承担只读展示，事件切换和保存不得修改管理员配置。非管理员计算方式必须允许逐事件切换该状态。
- 角色资料中的默认循环可用 `custom`、`single`、`sheer` 或 `anomaly` 描述其计算内容，但载入工作台后必须转换为运行态 `adminDefault`；只有这样，同一角色切换影画时才会自动刷新到对应的最高可用变体。
- 目标的防御、等级系数、初始失衡倍率和各元素抗性相互独立；目标配置不得继续保存全局 `stunned`，切换结算属性不得覆盖其他属性输入。旧非管理员存档按“事件显式值、旧目标值、默认 true”迁移，管理员默认循环始终以管理员事件配置为准。
- 默认 Buff、影画 Buff、用户手动取消项和 runtime 输入分别保存。Buff 弹窗只编辑草稿，确认后才应用，取消必须无副作用。
- 工作台自定义 Buff 只允许新建面向用户的精简白名单：六属性伤害加成必须分别保留；六属性版贯穿增伤、属性暴击伤害、属性无视防御、属性减抗和属性抗性无视不得出现在候选项中，改用通用或“当前属性”类型；“异常掌控超过 140 时每点转异常精通”不得向用户开放。维护端必须继续提供全部明确属性字段和阈值类型，计算核心及存量自定义 Buff 不得因用户白名单收窄而丢失兼容性。
- 每个已勾选的队友音擎 Buff 必须独立选择该音擎合法范围内的精修等级；新勾选和缺失等级默认精 1，切换等级立即重新物化效果预览但保留规则级 runtime 输入。取消勾选和批量移除同时清理等级引用与 runtime，批量添加建立精 1 引用，取消弹窗不得落库。
- 队友音擎等级以 `{ id: "wEngine:<engineId>.team", sourceCategory: "wEngine", sourceKind: "wEngineTeam", wEngineModificationLevel }` 轻量引用保存在 `combat.addedBuffs`，新保存必须同时保留 active ID 与引用；合法的旧 `addedBuffs`-only 引用应恢复为启用。加载和构建输入按对应音擎范围夹紧旧 UI 值，缺失或非数字按精 1；计算核心收到非法直接输入仍按严格契约回退默认档。
- 构建输入必须将引用投影为 `combatBuffs.wEngineTeamModificationLevels[buffId]`，工作台计算与优化器复用同一映射。已启用 Buff 摘要显示所选精修等级；自定义页、已选计数和“自定义 N 条”只统计 `sourceKind === "custom"`，不得把等级引用重复展示为自定义 Buff。
- 角色、音擎和 Buff 来源已有图片时必须显示真实资源；所有 catalog `/assets/...` 引用必须在构建时验证存在。

## 驱动盘、账号与本地数据

- 账号、当前账号、驱动盘、导入记录和套装预设继续使用既有 IndexedDB `zzz-calculator-user-store` 与 localStorage fallback `zzz-calculator.userStore.v1`，不迁移存储结构或键名。
- 旧构建选择键 `zzz-calculator.homeSelection.v1` 继续用于尽力恢复；新版自己的状态键不得破坏旧数据。
- 驱动盘仓库始终按当前账号隔离；手动 1-6 号位、套装预设和优化结果互不复制或串改底层驱动盘。
- 驱动盘导出必须包含当前账号的全部驱动盘且不受搜索、槽位或主词条筛选影响；不得包含其他账号或套装预设。原生文件使用带版本号的 `zzz-calculator-drive-disc-export` 格式，移除 `ownerId` 和派生指纹，回导时重新计算指纹并统一归入导入时的当前账号。
- 原生导出文件与扫描器 JSON 必须共用导入预览和确认流程。同账号同 ID 记录原位更新，完整重复回导不得新增副本，同内容但不同 ID 的原生仓库记录必须可以在空账号中完整恢复；未知原生版本和损坏结构必须明确拒绝，不能降级为扫描器数据解析。
- 扫描器导入必须先展示差异；“同步删除缺失盘”只能在确认流程中显式启用。删除驱动盘、套装和账号必须展示影响范围，当前账号不可直接删除。
- 扫描器本地/云客户端选择、Helper 连接、包准备进度、取消和导入结果必须保留；扫描 payload 与 manifest 由自动化测试锁定。

## 优化器

- 默认算法为 `exact-super-bound`，公开算法名、输入输出、进度和取消数据结构保持兼容。
- “严格精准”必须保持 Top 10 驱动盘 ID、稳定顺序和既有浮点误差契约，不得只保证第一名或近似分数。
- 标准、白盒、compiled、dense、固定套装评分、乐观上界、Node 优化器和浏览器 Worker 必须按候选套装计数识别定向 2 件套，并携带同一 `skillTags`；拂晓生花 2 件套只提升有技能来源的普通攻击，不提升终结技或手填倍率事件。
- 未限定额外 2 件套时只允许完整 4+2 和限定套装 6 件同套；4+1+1、5+1 不得进入预计组合数或结果。无合法计划时统一提示“已有驱动盘太少，无法组成 4+2 或 6 件同套。”，最低面板失败不得误用该提示。
- Node 适配器保留 `worker_threads` 并行；浏览器适配器只允许一个控制 Worker，禁止创建嵌套任务 Worker，并保留事件循环让步和准备阶段进度。历史 `exact-super-bound-parallel` 浏览器设置必须静默迁移为 `exact-super-bound`。
- 状态机必须区分准备、运行、取消中、已取消、完成和错误；进入取消中后不得保留可重复点击的取消操作。
- 固定套装专用评分器、套装计数内核缓存、目标相关词条淘汰和后缀前沿必须带安全回退；固定/自动 4+2 fixture、Node 并行、浏览器串行、历史设置迁移、进度、取消及 160-seed fuzz 必须保持通过。

## 维护功能

- 维护页覆盖角色、角色技能、音擎、驱动盘套装、异常/紊乱、队友 Buff、场地 Buff 和 Boss Buff 八类资源；`systemBuffs` 不在导航中显示。
- 八类资源必须使用结构化表单维护，普通界面、搜索结果和保存预览不得显示或编辑内部 ID，也不得恢复可编辑原始 JSON。角色技能维护器允许在默认折叠的“技术信息”中只读显示技能目录、招式和倍率行 ID，但不得提供编辑入口。
- 维护页保持顶部资源页签、左侧资料列表和右侧专用表单；不得重新合并为按字段名推断行为的通用递归编辑器。
- 角色默认计算方式必须区分“自定义”“最大化单个技能伤害/贯穿伤害”“最大化异常伤害”；技能倍率引用继续使用技能表、大类、招式和倍率行级联。
- 增幅对象必须明确区分“常规属性 / 全局效果”“异常目标”“指定角色招式”“通用技能大类”和“通用招式标签”。异常目标必须保存 `target.kind: "anomaly"`、属性异常/紊乱结算类型和至少一个具体异常；指定模式提供角色、大类、可选招式和可选倍率行，通用模式支持多选大类，标签模式支持多选冲刺攻击、强化特殊技和支援攻击。同一规则的标签按 OR 匹配，但不得与指定招式或通用大类混用。维护页与工作台自定义 Buff 不得显示或新建 `moveIdPrefixes`，也不得出现“既有招式系列”；工作台自定义 Buff 不得暴露缺少异常目标选择的 `anomalyDurationBonusSeconds`。
- 每个技能招式必须保存合法 `skillType`，可选 `skillTags` 只能使用显式白名单且同一招式不能重复；冲刺攻击继续属于 `dodge`，强化特殊技继续属于 `special`，支援攻击继续属于 `assist`。标签不得根据中文名称或 ID 推断。特殊技、闪避和支援技保持现有顶部粒度，连携技与终结技必须独立匹配。同一角色的一个类型不得拆到多个技能目录。正常事件生成和新目标匹配只能使用显式类型，不得根据 ID 前缀推断；已知旧前缀必须在工作台加载、运行时匹配和维护保存前自动迁移，未知前缀必须阻止维护保存并给出可操作错误。
- 维护页不得显示或新建旧“适用范围”、伤害类型、伤害属性或技能限定筛选。新保存的效果规则不得包含 `appliesTo`；技能范围只通过外层 `target.kind: "skill"` 表达，异常范围只通过外层 `target.kind: "anomaly"` 表达，属性限制使用明确的属性增伤、属性暴击伤害、属性无视防御、属性抗性无视或属性贯穿增伤类型，多属性效果拆成共享层数的多条规则。
- 驱动盘普通面板型 2 件套固定按局外面板效果校验；仅 `target.kind: "skill"` 的定向 2 件套规则按局内事件增幅校验，凑齐两件后自动注入且不需要活动 Buff ID。4 件套的 `selfBuff` 与 `teamBuff` 固定按局内 Buff 校验；维护草稿和 API 保存不得依赖显式 `scope` 绕过或触发范围限制，清洗后的驱动盘数据继续省略 `scope` 与 `appliesToOutOfCombatPanel`。无显式范围的 4 件套必须允许技能增幅、事件增幅和 Buff 修饰器；2 件套的非技能目标仍不得借伪造 `scope: "inCombat"` 写入事件规则。
- 效果规则可独立保存条件、持续时间、冷却时间、覆盖率、层数和 `requirement.specialty`；时间字段不得自动推算覆盖率，缺失覆盖率按 100% 计算，角色特性不匹配时只忽略受约束规则。团队效果的 `exclusiveGroup` 必须保证同组只结算一次，当前装备者的原始朋克效果优先于外部队友副本，忽略明细必须说明同名去重原因。
- 运行时必须继续计算旧 `appliesTo` 浏览器状态。维护草稿和 API 保存前应将可无损转换的元素筛选、技能目标及其修饰引用迁移到新结构；`damageKinds`、`anomalyEffects` 等无法无损转换的筛选必须保留读取兼容，但阻止维护落盘并提示重新选择明确增幅类型。
- `physicalCritDmg`、`fireCritDmg`、`iceCritDmg`、`electricCritDmg`、`etherCritDmg`、`windCritDmg` 只增加同属性直伤和贯穿伤害的暴击伤害；对应六种 `*DefIgnore` 只增加同属性事件的无视防御。两组属性类型必须覆盖标量、编译、稠密优化器和白盒说明，且不得影响其他属性或异常专用暴击乘区。
- 定向 `dmgBonus` 的维护文案统一为“技能目标伤害加成%”，进入增伤乘区；`skillMultiplierBonus` 统一为“技能倍率加算%”，直接加到技能倍率。维护页、工作台、玩家摘要和白盒说明不得混用这两个概念。
- 技能目标、效果规则、计算事件、Buff 修饰器和核心技强化各自使用稳定布局，不能按动态字段数量猜测列数。桌面编辑区和倍率表独立滚动，移动端业务字段单列且删除按钮不占字段轨道。
- 弹窗、抽屉和嵌套面板的表单必须按自身容器宽度响应，不得以整个 viewport 代替实际可用宽度。字段标签、当前值、校验提示和输入控件不得被裁切；密集列表主动省略时必须提供完整文本提示。
- 图片来源与 `sources[]` 必须分开编辑，多条资料来源增删和往返不得相互覆盖。音擎与驱动盘效果必须可维护现有条件、持续时间、分段文案、覆盖率和 Buff 修饰器，技术条件值在控件和预览中使用中文标签。
- 角色表单只显示管理员默认循环的紧凑摘要，具体配置使用影画页签与“左侧目标事件列表、右侧当前事件编辑”的主从式弹窗；技能组定义仍保持原有展开编辑方式。目标事件超过可见高度时必须能够滚动访问全部事件，不得被弹窗内容区裁切。弹窗取消不得修改角色草稿，应用后才标记维护草稿已变更。
- 角色默认循环的根配置必须固定为 `0 影`，附加方案只能使用未占用的 `1-6 影`并按等级排序。新增影画方案复制当前循环、重建全部事件 ID 并同步重映射 `selectedEventId`；目标事件使用可读事件摘要选择，删除非目标事件时保留仍然有效的目标，删除目标事件后选择相邻事件，最后一个事件不可删除。
- 叶瞬光管理员循环必须按影画解析：0-1影使用“0影叶12变2大”，2-5影使用“2影12变2大”，6影使用“6影叶12变6大”。6影固定包含2影短轴12次、逐云惊霆2次、斩妄开天6次和归尘6次；6影末击1500%倍率只作用于归尘与斩妄开天，并让斩妄开天继续继承2影无视防御。
- 共享层数组通过中文显示名选择或新建，复制时重建组标识并保持复制范围内的共享关系。
- 技能组的 `defaultCount/minCount/maxCount/step` 必须由系统固定为 `1/0/100/1`。覆盖率只能配置在单条 `effects[]` 规则上：维护页逐规则开启并以 `0%-100%` 维护 `default`，首页和优化器使用 `0-1` 小数输入，存储使用 `0-1`，`min/max/step` 固定为 `0/1/0.1`。未开启的规则必须按 100% 计算并忽略运行时注入；首页和优化器只为已开启规则提供独立输入。旧父级 catalog/runtime 覆盖率必须读取兼容并迁移到各子规则，新保存不得继续写父级字段。
- 新建请求允许省略顶层 ID；后端在校验前补齐顶层与嵌套 ID，已有 ID 必须保持不变。场地 Buff 继续根据玩法、版本、期数和名称生成稳定 ID。
- 异常资料读取 `anomalyEffects.effects` 并按 `settlementType` 区分异常和紊乱；队友 Buff 按角色分组、一次编辑一个 Buff，保存时保持“队友 ID + Buff ID”双层结构。
- 队友角色分组必须在角色层保存合法的 `attribute` 与 `specialty` 枚举，维护页新增或保存时两者均必填，`/api/catalog` 与 `/api/meta` 必须完整保留。队友 Buff 弹窗只在对应页签显示动态多选筛选；同维度取并集，属性与特性之间取交集，并继续与文本搜索叠加。批量添加和移除只作用于筛选后的可见 Buff；旧缺字段分组仅在未启用筛选时可见，每次重新打开弹窗重置筛选。
- 跨资源引用使用可搜索的可读名称选项，复制条目必须重建其拥有的嵌套 ID 并同步重写复制范围内引用。
- 八类资源均支持新增、复制、编辑、校验、保存、删除、删除确认、未保存离开提示和本地草稿恢复。
- 队友 Buff 同时支持删除当前 Buff 和确认后删除整个角色组；旧版 `zzz_maintenance_vue_draft_v1` 草稿必须可迁移到结构化草稿。
- 保存或删除成功后必须重新获取完整 `/api/maintenance/catalog`，不得用单文件响应覆盖组合 catalog。
- 写请求成功但完整目录刷新失败时，界面必须保留“已经写入”的身份，重试只能重新 GET 目录，不得重复 POST/DELETE；保存、删除期间禁止重复提交。
- catalog 文件的 read-modify-write 必须串行执行，并通过同目录临时文件原子替换；两个并发维护请求不得互相覆盖。
- 服务默认只监听 `127.0.0.1`。维护写入拒绝非 allowlist 的跨源 Origin，并覆盖 DNS rebinding 场景；公开域名必须通过 `MAINTENANCE_ALLOWED_ORIGINS` 明确授权。
- 维护关闭时 `/maintenance` 与 `/maintenance.html` 返回 404、维护 API 返回 403、导航隐藏入口；开启时 `/maintenance` 提供 Vue 页面，`/maintenance.html` 以 308 跳转到 `/maintenance`。

## 路由与发布

- SPA 正式路由为 `/`、`/discs` 和 `/accounts`；`/calculate.html`、`/drive-discs.html`、`/accounts.html` 继续跳转到对应 Vue 路由。
- Node 只服务 Vue 构建产物，不回退到旧静态前端。缺少产物时，开发服务必须给出明确构建提示。
- 非法 URL 百分号编码必须返回 400，不能终止服务进程；路径只允许解码一次。
- Pages 默认不生成维护页面，也不复制维护专用脚本或样式；部署工作流只由 `main` 更新触发。
- `dist/`、`output/`、`downloads/`、日志和本地用户数据不进入 Git；`examples/`、业务数据、源代码和回归测试必须保留。
